Evals before Features: What Building a Hinglish Voice Agent Taught Me about AI Product Roadmaps

Navajit D 2 June 2026 9 min read

A month ago, my former colleague Kaustubh and I won a voice AI buildathon by popular vote. Last weekend, he called to defend our title.

Out of six problem statements from the Vedantu team, we picked Post-Demo Conversion — the most complex one. The brief: build a voice agent that calls parents within 2–4 hours of a free demo class, gauges their child's experience, handles real-time objections (price, child feedback, decision loops), drives toward enrollment, and escalates to a human counselor when needed. I proposed extending the scope: capture structured sentiment and objection data from every call to feed CRM intelligence and agent self-improvement.

We called it Aria. She's a Hinglish voice AI who acts like a senior education counselor. Try her here.

This post is not an architecture tutorial. It's the part architecture tutorials don't cover.

Building the stack under pressure

Constraints showed up at every layer, even for a one-shot demo.

For STT, we needed accurate transcription, voice-activity detection, and noise cancellation — without paying ElevenLabs. Deepgram was the answer: best quality, generous free tier. For TTS, Sarvam has by far the most generous free tier and near-state-of-the-art Hindi voice quality. Telephony is typically the second-most-expensive component, but the Vedantu team covered it with a ₹1,000 credit from Vobiz.

The LLM layer was where it got interesting. We weren't paying OpenAI. So I proposed building a key-rotating proxy: the three of us — Kaustubh, Neelam, and I — each got Cerebras and Groq API keys, and the proxy rotated through them, failing over on quota errors mid-stream. The bot talks plain OpenAI Chat Completions to a local /llm endpoint, so swapping providers is a config change, not a code change. That flexibility turned out to matter.

None of us have been full-time engineers in the last two years. We built everything with Claude Code. The right way to use it: cover your requirements exhaustively in the first prompt, and it only comes back for clarifications. Front-load the thinking or you spend the session giving feedback and burning context.

The part that actually takes the day

Claude can scaffold a working voice agent in under half an hour — if you've procured all the API keys on time. A counselor agent that opens with context, classifies parent intent, runs an objection play, and closes: we had that framework within twenty minutes.

I had the prompt. I had the persona. I had the conversation flow.

But the hard parts — the ones that determine whether your demo lands or your user hangs up — don't announce themselves:

  • What happens when the LLM key rate-limits mid-call?
  • What happens when STT mishears a Hindi "हाँ" as an Odia "ହଁ"?
  • What happens when the parent says "अच्छा ठीक है" and the bot keeps talking instead of listening?
  • What happens when the model drifts on the child's name by turn 7 of a 15-minute call?

None of these are features. All of them are the existence of the product. Every one I had to discover the hard way, ship a fix, and add to my mental model of what AI shipping actually means.

The lesson I'd write on a stone tablet: for an AI product, the roadmap is evals → guardrails → fallbacks → telemetry → then features. PMs who run that order build differently from PMs who don't.

Specs become accuracy thresholds

In traditional software, a spec looks like: When the user clicks Submit, validate the form, write the row to the database, navigate to /confirmation. Deterministic in, deterministic out.

For Aria, the closest equivalent looked like:

"After each parent turn, classify intent as one of {positive, soft_no, ambiguous, hard_no} with confidence ≥ 0.7. If hard_no with confidence ≥ 0.8, respect it on the first signal — exit warmly, capture one reason if tone permits. If soft_no, identify the primary objection (price, comparison, child_experience, spouse, timing, trust, logistics) and route to the matching play. Maximum 3 objection cycles before offering a senior-counselor callback."

That's not a fixed-rule spec. It's an accuracy threshold, a fallback definition, and a state machine simultaneously. And you don't know whether it works until you have an eval.

Aria's eval runs a set of Hinglish conversation states — take a price objection as an example — across providers, measuring response speed and quality on every turn. The quality judgment is still human; the speed comparison is mechanical.

What the eval revealed mattered more than the rankings. It showed where the spec was underspecified. "Route to the matching objection play" sounds precise until you run the calls and watch the model classify "I need to discuss with my husband" as a logistics issue one time and a trust signal the next. The eval doesn't fix the ambiguity — but it makes the ambiguity visible, and visible is where product work starts.

We ended up with Groq llama-4-scout as the primary model — fastest, zero errors across test calls — dropping Cerebras when we found its models couldn't disable reasoning mode (on a voice agent, that means TTS reads the model's inner monologue aloud to the caller). Voice is unforgiving: a small per-turn latency handicap compounds across a 15-minute call. You can't roadmap features until you've benchmarked the substrate.

Three things learned the hard way

Latency doesn't come from where you expect. End-to-end from speech-end to first audio ran 0.95–1.35 seconds. Deepgram STT accounted for ~450ms; the LLM only ~100ms. TTS time-to-first-byte dominated the rest. One fix delivered disproportionate return: on an outbound call, the agent initiated it — so the agent should speak first. We had a 3-second "wait for the parent to say hello" timeout before TTS synthesis started. Removing it dropped dead air from ~5 seconds to 1–2. We also fire a throwaway single-token LLM completion the instant a call starts — the ~2-second TTS greeting hides the cold TLS handshake, so the parent's first reply lands as fast as any subsequent turn.

The language layer is its own product spec. Romanized Hindi ("Namaste, kaise hain") causes Sarvam to mispronounce everything. The model must emit Devanagari ("नमस्ते, कैसे हैं"); English brand names stay in Latin script. This is a hard, per-turn prompt rule, not a suggestion. Relatedly: a stray Odia or Tamil word from the transcriber that contradicts the flow is almost certainly a code-mix artifact, not the parent switching languages or registering a refusal. Tell the model to ignore it, not classify a hard_no. And backchannels — "अच्छा," "हाँ," "हम्म" mid-sentence — are encouragement, not interruptions. A minimum-words strategy (3+ words to interrupt while the agent is speaking; 1 when silent) keeps Aria from flinching at every nod.

"Connected" does not mean "working." ElevenLabs' Realtime WebSocket opened fine, then the server closed it within 0.5 seconds — quota exceeded. Pipecat's reconnect loop hid the failure: the bot spoke into a closed pipe; the parent heard silence. Our fallback only triggered on initialization failure, so it never engaged. Lesson: a health check that verifies only "the socket opened" is a lie. Watch for early server-side closes. Make fallbacks trigger on runtime failure, not just construction failure.

Memory is not the same as history

Pipecat keeps the full LLM context — every user message, every assistant reply — for the duration of the call. A 128k-token model can hold a 30-turn conversation a hundred times over. You'd assume the model can recall anything it said earlier.

It can. And it still drifts.

By turn 8, the model occasionally called the child "Arav" instead of "Aarav." By turn 11, it re-pitched an offer it had already made at turn 4. By turn 12, it re-ran a price question it had already asked. The history was there. The model just didn't reliably ground on it.

The fix: a structured CallMemory dataclass that sits on top of the LLM history, tracking objections already handled, whether the price offer was stated, the parent's exact name spelling from the CRM, any promises made. At the start of each turn, that block is rendered to text and prepended to the system prompt as ground truth. A few hundred bytes of extra context per turn. It eliminates roughly two-thirds of the drift.

The general principle: cheap and authoritative on top of expensive and probabilistic. If you can pin a fact deterministically, pin it. Don't ask the model to remember it.

Infrastructure decisions are product decisions

Building Aria surfaced a class of non-functional requirements I hadn't needed to think about in traditional product work — requirements that don't appear in any design doc template because they only exist when the system is probabilistic and real-time.

Per-turn response latency — not just "is it fast" but whether each extra millisecond compounds into a broken conversation across 30 turns (long-lived WebSockets on Cloud Run; pre-warmed proxy connections). Mid-call provider resilience — graceful degradation when a quota runs out during a call, not just at startup (key-rotating proxy with runtime failover; health checks that catch server-side closes, not just connection opens). Call-level state persistence — facts the system must track reliably, independent of model attention (Firestore for call records, since Cloud Run filesystems don't persist between instances). Secret hygiene at runtime — API keys that rotate without redeployment and don't leak into logs (Secret Manager, IAM-scoped service account — 12 stack traces before the IAM grant was right).

None of these were in the original brief. All of them had a user-facing consequence. In probabilistic, real-time systems, the line between an NFR and a product decision is mostly fictional.

What this build taught me about the role

A non-AI PM ships features. An AI PM ships features plus the fallback when the feature is wrong, plus the eval that proves it's right more often than yesterday, plus the telemetry that catches it drifting, plus the runbook for when the provider changes its rate limits at 2am.

The job is closer to building reliable spacecraft than building deterministic websites. Which is funny, because I trained for the spacecraft part first.

Aria is not a finished product. The demo is rate-limited to 3 calls per phone for cost reasons; there's a real list of unshipped work tracked in FUTURE.md. But it places real phone calls, runs an LLM you can actually hear, and I unlearned and relearned more about shipping AI products in a weekend building this than I could in ten AI PM webinars.
